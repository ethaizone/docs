The Art of the Name

A visual guide to writing clean, consistent, and maintainable code across your entire stack.

Why Naming Matters

The primary goal of any naming convention is to maximize clarity and minimize cognitive load. We write code for humans first, machines second. This guide establishes a unified standard to ensure our code is predictable, readable, and less prone to bugs.

Clarity Over Brevity

A long, descriptive name like `calculateTotalAmount` is always superior to a short, cryptic one like `calcTtl`. Your goal is to eliminate ambiguity.

Consistency is King

A reasonable, consistently applied convention is far better than a "perfect" but sporadically used one. Uniformity creates predictability.

Universal Truths: Naming Booleans

To make boolean values instantly recognizable and their purpose clear, we use specific prefixes. This convention applies globally across Go, PHP, and TypeScript.

is...

Describes a current state or quality. Answers: "What is it right now?"

isLoading, isVisible

has...

Indicates possession of a property. Answers: "Does it contain...?"

hasError, hasChildren

should...

Indicates a future or conditional action. Answers: "What is supposed to happen?"

shouldUpdate, shouldRetry

can...

Indicates an ability or permission. Answers: "Is it possible to...?"

canEdit, canDelete

A Language Showdown

While global patterns provide a foundation, we must respect the idiomatic conventions of each language. Here's a high-level comparison.

Philosophical Comparison

Key Takeaways

  • Go: Prioritizes simplicity and package visibility through casing. Brevity is encouraged in small scopes.
  • PHP: Relies on the strict, community-driven PSR standards for interoperability and professionalism. Naming is verbose and explicit.
  • TypeScript: Focuses on a clear separation between type-level constructs (PascalCase) and runtime values (camelCase), enforced by powerful tooling.

Spotlight on React: Props vs. Handlers

React conventions create a clear data flow. A critical pattern is the distinction between props passed down (`on...`) and handlers defined within a component (`handle...`).

Parent Component

const handleNameChange = () => { ... };

// The handler implements the logic
return <Input onChange={handleNameChange} />;
// It's passed to an 'on' prop

→
↓

Child Component's API

The `on` prefix defines the public API for events the child can emit.

props: { onChange: Function }

The "Why"

This separation makes component interaction predictable.
`handle...` = "What to do" (Implementation).
`on...` = "What happened" (API/Event).
